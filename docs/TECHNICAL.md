# immich-auto-dumper ‚Äî technical documentation

This document describes how the script works internally. For installation and daily
use, see the [README](../README.md).

## Design principles

- **No privileges.** Everything runs as the invoking user ‚Äî never `sudo`. Docker access
  comes from `docker` group membership; files owned by the container's user are deleted
  *through* the container.
- **Immich is the source of truth.** The tool reads Immich's database to decide what to
  do and performs one narrow write (relocating an asset). It never "repairs" Immich: on
  any inconsistency it stops and asks the user to fix things on the Immich side.
- **Refuse rather than guess.** Every run starts with pre-flight checks (Docker access,
  DB schema, storage identity, path consistency, recent DB backup). Any failure aborts
  the run before the first destructive operation.
- **Idempotent and resumable.** A run interrupted at any point leaves each asset either
  fully archived or fully intact; the next run picks up where it left off.

## Components

```
immich-auto-dumper.sh    entry point: CLI parsing, setup wizard, status, cron start/stop
lib/utils.sh             logging, locking, prereqs, storage marker, disk measurements
lib/ui.sh                interactive prompts (whiptail with plain-text fallback)
lib/detect.sh            auto-detection of containers, DB credentials, mounts
lib/db.sh                every SQL statement in the project (psql via docker exec)
lib/archive.sh           the archiving engine (candidate loop + per-file pipeline)
lib/backup_db.sh         DB-dump mirroring with retention
cron/crontab.example     cron template instantiated by `start`
config.conf              generated by setup; sourced at launch (git-ignored)
```

The entry point resolves itself through the `~/.local/bin` symlink (`readlink -f`) so
`lib/`, `config.conf` and `cron/` are always found in the real install directory.

## Path model

Three coordinate systems describe the same file:

| Space | Example | Used by |
|---|---|---|
| Host, internal library | `/home/x/immich-app/library/library/admin/2024/06/a.jpg` | `cp`, `du`, `stat` |
| Container / DB, internal | `/data/library/admin/2024/06/a.jpg` (`IMMICH_DB_LIBRARY_PREFIX`) | `asset.originalPath` before archiving |
| Container / DB, archive | `/kdrive/Alice/2024/06/a.jpg` (`ARCHIVE_CONTAINER_PATH`) | `asset.originalPath` after archiving |

`lib/archive.sh` converts between them by prefix substitution
(`db_path_to_host_path`, `host_path_to_db_path`, `archive_build_dest_path`). The
destination preserves the full sub-path after the user folder, so any Immich storage
template layout survives unchanged. `USER_MAP` in `config.conf` renames the per-user
top-level folder on the external storage.

## Database access

All SQL goes through `_db_exec` (`lib/db.sh`), which runs `psql -t -A` inside the
Postgres container. Two details matter:

- stdin is redirected from `/dev/null` ‚Äî `docker exec -i` would otherwise drain the
  `while read` loop feeding the archiving engine.
- Three escaping layers are applied depending on context: `_db_escape` (SQL string
  literals), `_db_escape_like` (LIKE wildcards), `_db_escape_regex` (POSIX regex
  metacharacters). File names containing quotes, `%`, `_` or regex tokens are handled
  literally.

### Schema validation

`db_check_schema` runs before any operation (via `check_prereqs`) and verifies against
`information_schema` that every column the tool relies on exists:

- `asset`: `id`, `originalPath`, `isOffline`, `isExternal`, `libraryId`, `deletedAt`,
  `ownerId`, `visibility`, `fileCreatedAt`
- `asset_exif`: `assetId`, `fileSizeInByte`
- `library`: `id`, `ownerId`, `importPaths`, `deletedAt`

If an Immich upgrade renames or removes any of them, every run aborts at pre-flight with
an explicit error ‚Äî a clean refusal, never a half-run against wrong assumptions. The
running Immich version (from the image's `IMMICH_SOURCE_REF`, falling back to
`IMMICH_VERSION`) is logged at the start of each run so a failure can be tied to the
upgrade that introduced it. Limits to be aware of: the check verifies column *presence*,
so purely additive migrations or semantic changes with unchanged names are not detected.
After each Immich upgrade, run `immich-auto-dumper test_run` before re-enabling the cron.

## The archiving engine (`archive_run`)

### Pre-flight

1. `check_prereqs` ‚Äî Docker reachable as this user, `bc` present, schema valid.
2. `check_archive_dest_ready` ‚Äî storage marker present and matching (see below).
3. `guard_path_consistency` ‚Äî see below; on failure the cron is disabled.
4. `acquire_lock` ‚Äî PID lock in `/tmp`; stale locks (dead PID) are cleaned up.
5. Boundary check ‚Äî `TARGET` (and `MAX` for unforced runs) must be configured.
6. **Backup precondition** ‚Äî a DB dump younger than 7 days must exist in
   `UPLOAD_LOCATION/backups/`, or the run aborts (archiving rewrites `originalPath`
   rows; a fresh dump is the safety net).

### Trigger and candidate selection

The trigger compares the **measured size of `UPLOAD_LOCATION/library`** (`du -sb`,
apparent size) with the configured boundaries ‚Äî unrelated data on the same filesystem
never influences it. Normal runs archive only when the size exceeds `MAX`; `--force`
runs archive regardless. Both bring the library down to `TARGET`, never below.

`db_get_archive_candidates` groups active assets (`deletedAt IS NULL`, not offline, not
already external) by their **immediate parent directory** and orders groups by the
oldest `fileCreatedAt` they contain. This makes the unit of work template-agnostic: the
genuinely oldest photos leave first, and a directory is never left half-archived ‚Äî the
freed-bytes threshold is only evaluated between directories.

### Per-file pipeline (`_archive_move_file`)

For each asset in a candidate directory:

1. **Destination conflict check.** If the destination already exists with the same
   size, the file is treated as already archived (DB update only, no copy). A different
   size is an error ‚Äî the asset is skipped.
2. **Copy + verify.** `cp` to the external storage; the exit code is checked and the
   destination size must equal the source size. A partial copy (full disk, dying mount)
   is deleted and the asset skipped.
3. **DB update + library adoption.** One atomic `UPDATE` rewrites `originalPath` and
   *adopts* the asset into the external library whose `importPaths` covers the new path
   and whose owner matches the asset's owner (`libraryId` set, `isExternal = true`,
   `isOffline = false`). Adoption is what prevents Immich's periodic library scan from
   re-importing the file as a duplicate: the scan matches on `(libraryId, originalPath)`
   and finds the row already there. If **no** external library covers the destination,
   the update is rolled back and the asset skipped ‚Äî archiving without adoption would
   guarantee duplicates at the next scan. Dry runs surface this case as a warning
   (`db_asset_would_be_external`).
4. **Container-side read-back.** The file must be reachable at its new path from inside
   the Immich server container (`test -f`). If not, the DB row is reverted and the copy
   removed.
5. **Source deletion through the container.** Library files belong to the container's
   user, so `rm` is executed inside the container. A failure here is non-fatal: the
   asset already points at the destination; the source merely lingers as an orphan.

XMP/JSON sidecars (`x.jpg.xmp`, `x.xmp`, `.json` variants) are moved alongside their
asset; they are not tracked in the DB, so this is filesystem-only.

The rollback direction reuses the same `UPDATE`: pointing the path back at the internal
library matches no external library, which resets `libraryId`/`isExternal` to the plain
upload-asset state ‚Äî adoption is symmetric by construction.

## Storage identity and availability

At setup, a marker file (`.immich-auto-dumper.id`, a random UUID also stored in
`config.conf` as `ARCHIVE_STORAGE_ID`) is written at the root of the external storage.
Every run re-reads it (with a 10 s timeout to survive dead FUSE mounts):

- **Missing marker** ‚Üí the storage is treated as absent (an unmounted mount point is
  just an empty directory) and the run is skipped without writing anything.
- **Wrong marker** ‚Üí a different volume is mounted there; the run is refused.
- Marker present and matching ‚Üí proceed.

Because the marker travels with the data, moving the storage to a new host path only
requires re-running `setup`.

## Path-consistency guard (read-only)

`db_check_path_consistency` detects that Immich's view diverged from the config:

- the internal library prefix derived from a live asset no longer matches
  `IMMICH_DB_LIBRARY_PREFIX`, or
- archived assets under `ARCHIVE_CONTAINER_PATH` are reported **offline** while the
  storage is reachable ‚Äî the external library path was likely changed in Immich.

On inconsistency the run aborts and the cron entries are commented out
(`disable_cron`), so a stale configuration cannot keep acting. The user fixes the path
in Immich and re-runs `setup`. The tool never rewrites the DB to reconcile.

## DB-dump mirroring (`backup_db_run`)

Copies every non-hidden file from `UPLOAD_LOCATION/backups/` to
`<storage>/.immich-backup/`, then applies retention (`BACKUP_RETENTION`, newest kept by
mtime). It intentionally skips the consistency guard: mirroring dumps stays safe and
useful even while a path change is being resolved. It never touches the database.

## Scheduling

`start` instantiates `cron/crontab.example` into the user's crontab (via the
`~/.local/bin` symlink so paths survive updates): `dump_now` daily at 02:00, `sync_now`
weekly on Sunday at 03:00, both appending to `cron.log`. `stop` comments the entries out
and waits for a running operation to release the lock. The guard above may also disable
the entries autonomously.

## Logging

`_log` writes to stdout/stderr (colored on a TTY) and appends to
`$LOG_DIR/immich-auto-dumper.log` (default `~/.local/state/immich-auto-dumper/`,
honoring `$XDG_STATE_HOME`), truncated to `LOG_MAX_LINES`. File-logging failures never
abort a run.

## Failure-mode summary

| Event | Behavior |
|---|---|
| Storage unplugged / mount down | Run skipped, nothing written, resumes next time |
| Wrong disk mounted at the path | Run refused (marker mismatch) |
| Partial copy (disk full, I/O error) | Destination removed, asset skipped, source intact |
| No external library covers a destination | DB rolled back, asset skipped, warning in dry run |
| Immich schema changed by an upgrade | Every run aborts at pre-flight with the missing columns listed |
| External library path changed in Immich | Run aborts, cron disabled until re-setup |
| No recent DB backup | Archiving refuses to start |
| Concurrent invocation | Second run exits on the PID lock |
| Crash mid-run | Stale lock auto-cleaned; each asset is either fully archived or intact |
